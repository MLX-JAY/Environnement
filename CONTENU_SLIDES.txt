--- ASPECT TECHNIQUE 1 : Le Pattern Visitor pour la génération d'événements ---

L'architecture utilise un GenerateurEvenementVisitor pour gérer la communication
entre les biomes et les événements de manière découplée.

Au lieu d'avoir une logique codée en dur dans chaque classe de biome, chaque biome
implémente une méthode accept(BiomeVisitor<T>).

Ce pattern permet au visiteur de retourner dynamiquement des probabilités d'événements
spécifiques selon le type de biome visité :

- Forêt     → Purification (2.5%), PluieBénite (2.5%)
- Désert    → VentChaud (5%), Zephyr (3%), Tornade (1%)
- Mer       → Pluie (8%), Orage (4%), Tonnerre (2%)
- Ville     → Pollution (5%), Smog (3%), NuageToxique (1%)
- Village   → Pollution (5%)
- Banquise  → VentFroid (6%), Grèle (2%)
- Montagne  → Aucun événement

Comment ça marche dans le code (GenerateurEvenementVisitor.java:31-49) :

    @Override
    public Evenement visit(Foret foret) {
        if (random.nextDouble() < ConfigurationCreationEvenement.PROBABILITE_PURIFICATION_PAR_FORET) {
            Bloc position = foret.getPosition();
            List<Evenement> liste = factory.creerPurificationMultiple(position);
            if (liste != null && !liste.isEmpty()) {
                return liste.get(0);
            }
        }
        return null;
    }

Le biome se "fait visitar" et retourne l'événement généré via sa méthode accept().

--- ASPECT TECHNIQUE 2 : Le système de transformation séquentiel (Strategy) ---

Le moteur intègre 8 règles de transformation qui évaluent l'évolution des biomes
en fonction de 4 propriétés internes :

    - Température
    - Humidité
    - Pollution
    - Purification

Chacune des 8 règles possède des seuils précis (ConfigurationTransformation.java) :

1. Inondation → Mer
   - temp > 5, humidite >= 90, pollution <= 40

2. Desertification → Désert
   - temp > 50, humidite < 10 OU >= 90, pollution < 10

3. Glaciation → Banquise
   - temp <= 0, humidite > 20, pollution <= 20
   - CONDITION: doit être Mer (seules les mers gèlent)

4. Forestation → Forêt
   - temp 10-35, humidite 60-90, pollution < 25, purification >= 15

5. Civilisation → Village
   - temp 15-40, humidite 20-80, pollution <= 10, purification >= 30

6. Érosion → Montagne
   - humidite < 40, pollution <= 5, purification >= 40

7. Densification → Ville
   - temp 18-45, humidite 30-70, pollution 10-60, purification >= 15
   - CONDITION: doit être Village

8. PollutionExtreme → Désert
   - temp > 80, humidite < 10 OU >= 90, pollution < 10

La mécanique d'évaluation est optimisée : les règles sont testées dans l'ordre et la
première condition qui valide un "match" l'emporte, interrompant la boucle de
vérification pour ce tour (ManageurBasique.java, ligne ~290 : break).

--- ASPECT TECHNIQUE 1 : Le GestionDeplacementVisitor comme moteur physique ---

Ce visiteur agit comme un véritable moteur physique pour les événements climatiques.
Le code est centralisé dans une seule méthode deplacerEvenementMobile(), ce qui
rend la maintenance beaucoup plus simple.

Trois mécanismes assurent un déplacement naturel :

1. Vecteurs de déplacement persistants
   Chaque événement conserve en mémoire sa direction de base (directionX et directionY).
   Cela évite les changements de mouvement brusques ou erratiques.

   Dans le code (GestionDeplacementVisitor.java:156-167) :

       if (dx != 0 || dy != 0) {
           // Utiliser la direction existante si elle existe, sinon aléatoire
           dx = evenement.getDirectionX();
           dy = evenement.getDirectionY();
       } else {
           int dirIndex = random.nextInt(ConfigurationDirection.DIRECTIONS.length);
           dx = ConfigurationDirection.DIRECTIONS[dirIndex][0];
           dy = ConfigurationDirection.DIRECTIONS[dirIndex][1];
           evenement.setDirection(dx, dy);
       }

2. Analyse de la topologie (L'obstacle Montagne)
   Avant chaque déplacement, le GestionDeplacementVisitor vérifie la nature du biome cible.
   Si la case visée est une Montagne, l'événement ne la traverse pas. Le visiteur
   intercepte le mouvement et génère automatiquement une nouvelle direction aléatoire.

   Dans le code (GestionDeplacementVisitor.java:188-208) :

       if (biomeDestination instanceof Montagne) {
           // Si montagne, changer de direction
           int dirIndex = random.nextInt(ConfigurationDirection.DIRECTIONS.length);
           dx = ConfigurationDirection.DIRECTIONS[dirIndex][0];
           dy = ConfigurationDirection.DIRECTIONS[dirIndex][1];
           evenement.setDirection(dx, dy);
           // ... Retry avec la nouvelle direction
       }

3. Gestion des limites (Effet de rebond)
   Si la trajectoire amends l'événement à sortir des bords de la carte,
   le visiteur recalcule le trajet en inversant la direction, ce qui crée
   un effet de rebond géométrique.

   Dans le code (GestionDeplacementVisitor.java:172-183) :

       if (!carte.estCoordonneeValide(nouvelleX, nouvelleY)) {
           // Si hors carte, changer de direction
           int dirIndex = random.nextInt(ConfigurationDirection.DIRECTIONS.length);
           dx = ConfigurationDirection.DIRECTIONS[dirIndex][0];
           dy = ConfigurationDirection.DIRECTIONS[dirIndex][1];
           nouvelleX = position.getX() + dx;
           nouvelleY = position.getY() + dy;
       }

--- ASPECT TECHNIQUE 2 : Le Système d'Animation Fluide (Interpolation Temporelle) ---

Le programme sépare intelligemment la "logique de jeu" (qui avance tour par tour)
du "rendu visuel" (qui doit être fluide pour l'œil).

1. Indépendance du framerate
   Le système d'animation n'est pas basé sur le nombre de frames par seconde,
   mais sur une interpolation temporelle stricte. Le programme utilise
   l'horloge interne de la machine avec System.currentTimeMillis() pour
   se repérer dans le temps.

2. Calcul de la progression
   Le moteur calcule l'avancement exact de l'animation avec une formule précise :

       progressionAnimation = (tempsActuel - tempsDebutAnimationMs) / dureeAnimationMs

   Dans Evenement.java:36-44 :

       private double calculerProgressionAnimationActuelle() {
           if (progressionAnimation >= 1.0) {
               return 1.0;
           }
           long tempsEcouleMs = System.currentTimeMillis() - tempsDebutAnimationMs;
           double duree = Math.max(1L, dureeAnimationMs);
           return Math.min(1.0, tempsEcouleMs / duree);
       }

3. Déplacement au pixel près
   Grâce à cette progression (qui va de 0.0 à 1.0), le moteur de rendu calcule
   la position intermédiaire exacte de l'événement entre sa case de départ et
   sa case cible. La formule appliquée est :

       positionAnimationX = positionDepartAnimationX +
           (positionCibleAnimationX - positionDepartAnimationX) * progressionAnimation

   Cela garantit un glissement parfait de l'animation à l'écran, peu importe
   la puissance de l'ordinateur qui fait tourner le programme.

   Dans Evenement.java:25-34, les coordonnées fluides sont stockées :

       // Coordonnées fluides pour l'animation
       private double positionAnimationX;
       private double positionAnimationY;
       private double positionDepartAnimationX;
       private double positionDepartAnimationY;
       private double positionCibleAnimationX;
       private double positionCibleAnimationY;
       private double progressionAnimation = 0.0;
       private long tempsDebutAnimationMs;
       private long dureeAnimationMs;

   Le constructeur initialise ces valeurs (Evenement.java:59-68) :

       this.positionDepartAnimationX = position.getX();
       this.positionDepartAnimationY = position.getY();
       this.positionAnimationX = position.getX();
       this.positionAnimationY = position.getY();
       this.positionCibleAnimationX = position.getX();
       this.positionCibleAnimationY = position.getY();
       this.progressionAnimation = 1.0;
       this.tempsDebutAnimationMs = System.currentTimeMillis();
       this.dureeAnimationMs = Math.max(16, GameConfiguration.VITESSE_JEU);
